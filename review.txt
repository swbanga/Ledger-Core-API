# LEDGERCORE V1.0 MASTER BLUEPRINT AUDIT REPORT
## COMPLIANCE EVALUATION – GOVERNED BY THE READ‑ONLY ENTERPRISE REVIEW DIRECTIVE

### 1. THE ON‑POINT (Acknowledgments)

* **Append‑Only Ledger.**  
  `LedgerTransaction` and `LedgerEntry` are never updated or deleted after the transaction is persisted.  
  Entries are added once, `Post()` transitions status to `Posted`, and no code performs destructive changes thereafter.  
  This is the bedrock of an institutional‑grade ledger.

* **Double‑Entry Invariant.**  
  `LedgerTransaction.Post()` enforces `balance == 0`, forbids zero‑amount entries, and rejects mixed‑currency transactions.  
  The invariant protects the absolute mathematical integrity of the ledger.

* **Domain Purity (mostly).**  
  The domain contains rich entities (`Account`, `LedgerTransaction`, `LedgerEntry`) and strongly‑typed Value Objects (`Money`, `CurrencyCode`, `AccountNumber`, `AuditMetadata`).  
  Each Value Object carries its own validation – `AccountNumber.CreateUserAccount` enforces 10‑digit/zero‑start format, `Money` enforces same‑currency arithmetic.

* **CQRS Separation.**  
  Commands (`CreateAccountCommand`, `TransferFundsCommand`) and queries (`GetAccountBalanceQuery`) live in distinct feature folders.  
  Handlers follow the single‑responsibility pattern, and the use of `ISender` from MediatR keeps the controller slim.

* **Pipeline Behaviors.**  
  `LoggingBehavior`, `IdempotencyBehavior`, `KycLimitBehavior`, and `VelocityCheckBehavior` are wired into the MediatR pipeline.  
  Each behavior tackles a cross‑cutting concern without touching the core business logic – a textbook implementation.

* **Concurrency Token (RowVersion).**  
  `Account` and `AccountBalanceState` (if retained) carry a `byte[] RowVersion` column, providing optimistic‑concurrency guarantees with EF Core.

* **JWT Authentication & Authorization.**  
  `AuthController` issues JWTs with `JwtProvider`, and controllers combine `[Authorize]` with role‑based policies (`SysAdmin`, `Agent`, `User`).  
  This correctly gates administrative endpoints.

* **Integration Tests.**  
  `TransferFundsIntegrationTests` exercise the happy‑path, insufficient‑funds, zero‑amount rejection, KYC‑limit, and concurrency‑driven overdraft prevention.  
  They demonstrate a solid testing culture.

### 2. THE CRITICAL GAPS (Flaws & Missing Code)

#### (A) OUTBOX DUALITY – SPLIT BRAIN IN MESSAGE DELIVERY

Two conflicting outbox processors co‑exist:  
- `OutboxProcessorService` (MediatR `IPublisher`)  
- `OutboxProcessorJob` (MassTransit `IPublishEndpoint`)  

Only one can be active at runtime.  
The codebase ships both, causing *ambiguity* around which pathway actually delivers domain events to projection consumers.  
If the active runtime uses `OutboxProcessorService`, the MassTransit‑based `BalanceProjectionConsumer` will never receive an event.  
Conversely, if `OutboxProcessorJob` is active, the MediatR `AccountBalanceProjector` will never fire.  
This split brain makes the projection layer non‑deterministic – a critical architectural defect.

#### (B) MUTABLE BALANCE SNAPSHOTS VIOLATE “SINGLE SOURCE OF TRUTH”

The LedgerCore Blueprint explicitly demands that balances be **dynamically projected** from the append‑only ledger (the `LedgerEntries` table).  
Instead the current design stores balances in *two* mutable tables:

```
Domain.Projections.AccountBalance          (table AccountBalances)
Domain.ReadModels.AccountBalanceState      (table AccountBalanceStates)
```

These tables are updated by `BalanceProjectionConsumer` (or `AccountBalanceProjector`) whenever a `TransactionPostedDomainEvent` arrives.  
Hence:

- The system has **three** sources of truth: the ledger, the projection table, and the read‑model table.  
- Any temporary outage of the projection pipeline creates permanent drift that can only be repaired by a full replay – a massive operational risk.  
- Query handlers (`GetAccountBalanceQueryHandler`) read the snapshot instead of computing on‑the‑fly, directly violating the “append‑only” and “dynamic projection” tenets.

#### (C) DOMAIN LAYER DEPENDS ON MEDIATR

`IDomainEvent` inherits from `MediatR.INotification`:

```csharp
public interface IDomainEvent : INotification { }
```

Adding a third‑party framework dependency to the Domain layer breaks the **Domain‑Driven Design rule: the Domain must have zero external dependencies**.  
This contaminates the Domain with infrastructure/application concerns and prevents a pure clean‑architecture separation.

#### (D) DOUBLE‑ENTRY PHYSICS: NO MINIMUM ENTRY COUNT

`LedgerTransaction.Post()` performs these checks:  
- Sum of all entries equals zero  
- No entry has amount == 0  
- All entries share the same currency  

It does *not* require at least **2** entries.  
A transaction with zero entries will pass (sum-of-entries on an empty collection is zero, and the zero‑amount guard does not apply).  
A transaction with a single entry and a non‑zero amount is caught by the balance imbalance check, but the zero‑entry scenario remains open.  
This is a subtle but severe breach of double‑entry accounting – zero‑entry “ghost” transactions corrupt the ledger.

#### (E) JWT SECURITY WEAKNESSES

* **Token expiration computes an ambiguous `DateTime`.**  
  `_timeProvider.GetUtcNow().AddMinutes(...).DateTime` discards the time‑zone offset, potentially causing UTC‑vs‑local discrepancies during validation.  
  The industry standard is to use `.UtcDateTime` or `DateTimeOffset` consistently.

* **Secret entropy not enforced.**  
  `JwtSettings.Secret` is used directly as an HMAC‑SHA256 key.  
  No validation exists for minimum key length (should be ≥ 32 bytes).  
  A weak secret permits brute‑force forging of valid tokens – a high‑severity vulnerability.

#### (F) CQRS QUERY HANDLER VIOLATION

The `GetAccountBalanceQuery` (in `LedgerCore.Application/Features/Accounts/Queries/GetAccountBalance`) likely returns a stored snapshot rather than computing the balance on‑the‑fly from `LedgerEntries`.  
The presence of the `AccountBalances` DbSet in `IApplicationDbContext` confirms the design intent to read from a mutable projection.  
This contradicts the “dynamic projection” principle and the single‑source‑of‑truth requirement.

#### (G) OUTBOX TYPE‑RESOLUTION RISK

`OutboxProcessorService` resolves types via `Type.GetType(msg.Type)`.  
If the `InsertOutboxMessagesInterceptor` stores only the short type name (e.g., `"TransactionPostedDomainEvent"`) instead of the assembly‑qualified name, `Type.GetType` will return `null`, causing every outbox message to be silently discarded.  
Correctness depends on implementation detail that is not visible in the provided snapshot.

### 3. THE DEBT LEDGER

| #  | Debt Item | Severity | Notes |
|----|-----------|----------|-------|
| 1  | **Duplicate Outbox Processor (`OutboxProcessorJob`)** | High | Dead code, creates operational confusion. Must be removed. |
| 2  | **Race condition in IdempotencyBehavior (Redis).** | High | `GetStringAsync` followed by `SetStringAsync` is not atomic; two concurrent identical‑key requests can both pass the guard, defeating idempotency. |
| 3  | **Race condition in KYC and Velocity behaviors (Redis).** | High | Read‑modify‑write on `DistributedCache` without atomic `INCR`/`SETNX` allows limit bypass under concurrent load. |
| 4  | **No dead‑letter handling for outbox.** | Medium | A permanently failing message is retried indefinitely without alerting, consuming resources and masking other messages. |
| 5  | **Concurrency handling gaps in read projections.** | Medium | `BalanceProjectionConsumer` updates `AccountBalanceState` without explicit transaction retry logic; while RowVersion offers EF‑side concurrency, the code catches `DbUpdateConcurrencyException` only in tests, not in production. |
| 6  | **LedgerTransaction and LedgerEntry lack RowVersion.** | Low | Acceptable for append‑only design, but any future mutation (e.g., reversal) would need protection. |
| 7  | **LoggingBehavior at Info level includes no payload details.** | Low | Audit‑grade system demands that command parameters (excluding PII) be logged for traceability. |
| 8  | **Test fixture overrides Idempotency with FakeIdempotencyService**, effectively bypassing idempotency validation in integration tests. | Medium | Tests should exercise the real idempotency behavior at least once. |

### 4. EXECUTION ROADMAP (Priority Ordered)

> Each step assumes the current codebase is frozen.  
> The objective is to reach a **9.5/10 Enterprise Readiness Score**.

1. **REMOVE OutboxProcessorJob AND UNIFY OUTBOX FLOW.**  
   - Delete `OutboxProcessorJob` (and any MassTransit consumer not intended for production).  
   - Ensure only `OutboxProcessorService` publishes events, and that all projection handlers subscribe via MediatR.  
   - Verify that `InsertOutboxMessagesInterceptor` stores the **full assembly‑qualified type name** so that `Type.GetType(msg.Type)` succeeds.

2. **ELIMINATE MUTABLE BALANCE TABLES.**  
   - Drop `AccountBalances` and `AccountBalanceStates` tables and their corresponding ORM configurations.  
   - Remove `AccountBalanceProjector`, `BalanceProjectionConsumer`, `AccountBalanceState`, and `AccountBalance` C# classes.  
   - Refactor `GetAccountBalanceQueryHandler` to compute the balance by summing `LedgerEntries` for the given `AccountId` – **one source of truth, dynamically projected**.  
   - If a read‑model is truly necessary for performance, implement a *write‑model* replay mechanism that regenerates the snapshot from the immutable ledger at known check‑points, not live updates.

3. **ENFORCE MINIMUM ENTRY COUNT IN DOMAIN.**  
   - In `LedgerTransaction.Post()`, add `if (_entries.Count < 2) throw new InvalidOperationException(...)`.  
   - Update the integration tests to verify this constraint.

4. **DECOUPLE DOMAIN FROM MEDIATR.**  
   - Remove the `INotification` inheritance from `IDomainEvent`.  
   - Define a project‑internal `IDomainEvent` (or `IEvent`) in `Domain`.  
   - Introduce a MediatR adapter (e.g., `DomainEventNotification<T> : INotification`) in the Application or Infrastructure layer.  
   - Update the outbox interceptor and processor accordingly so that domain events are correctly wrapped and published.

5. **HARDEN JWT SECURITY.**  
   - Add startup validation: guarantee `_jwtSettings.Secret.Length >= 32`.  
   - Use `_timeProvider.GetUtcNow().AddMinutes(…).UtcDateTime` to produce UTC‑kind `DateTime` exactly.  
   - Consider adopting `SecurityTokenDescriptor` for cleaner API.

6. **FIX RACE CONDITIONS IN IDEMPOTENCY, KYC, AND VELOCITY BEHAVIORS.**  
   - Replace `IDistributedCache` usage with atomic Redis primitives.  
   - Implement a `SET` with `When.NotExists` for idempotency creation.  
   - For volume tracking, execute `INCR` atomically and compare the result against the limit in a Lua script (or transaction) to eliminate the read‑modify‑write window.  
   - If the project relies exclusively on `IDistributedCache` and cannot use raw Redis, evaluate alternative distributed‑locking mechanisms (e.g., distributed lock via RedLock).

7. **INTRODUCE DEAD‑LETTER HANDLING FOR OUTBOX.**  
   - Add an `Error` column (already present) and a retry count.  
   - In `OutboxProcessorService`, after a configurable number of failures, mark the message as dead‑letter and raise an alarm.  
   - Ensure the processing loop continues with other messages after a dead‑letter is identified.

8. **EXTEND INTEGRATION‑TEST COVERAGE.**  
   - Add a test that validates the idempotency behavior with actual `RedisIdempotencyService`.  
   - Add a concurrency‑stress test for KYC limit enforcement that proves the atomic behavior after the Redis changes.  
   - Add a test proving that `LedgerTransaction.Post()` rejects zero‑entry or single‑entry transactions.

9. **ENHANCE OBSERVABILITY.**  
   - Add structured logging of command payloads (excluding secrets/PII) at `Debug` level, plus mandatory `Information`‑level success/failure traces.  
   - Expose health‑check endpoints for Redis connectivity, database connectivity, and outbox backlog size.

**DELIVERABLE**: Implement the steps above sequentially.  
Upon completion, the system will conform to the V1.0 Master Blueprint in all dimensions and achieve the **Enterprise‑Ready** stamp.
